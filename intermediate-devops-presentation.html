Day 1

CI/CD and Basic Pipelines

What is CI/CD?

Continuous Integration (CI)

  • Developers frequently merge code changes into a shared repository
  • Each integration triggers automated builds and tests
  • Detects integration issues early
  • Ensures code quality through automation

Continuous Delivery/Deployment (CD)

  • Automates release process after CI
  • Delivery: Automated release to staging
  • Deployment: Automated release to production
  • Reduces manual intervention and human error

Key Principle: "If it hurts, do it more often" - Frequent small releases are safer than infrequent large ones

Benefits of CI/CD

Speed

  • Faster time to market
  • Rapid feedback loops
  • Quick bug detection

Quality

  • Automated testing
  • Consistent builds
  • Reduced human error

Reliability

  • Repeatable processes
  • Version control
  • Rollback capability

Business Impact

  • 46x more frequent deployments (DORA research)
  • 440x faster lead time from commit to deploy
  • 5x lower change failure rate
  • 170x faster mean time to recovery

Pipeline Stages

A typical CI/CD pipeline consists of these core stages:

1. Source

Code commit triggers pipeline

2. Build

Compile code, create artifacts

3. Test

Run automated tests

4. Deploy

Release to environment

Stage Details

  • Source: Version control (Git), webhooks, branch strategies
  • Build: Dependency installation, compilation, artifact creation
  • Test: Unit tests, integration tests, security scans
  • Deploy: Staging, production, blue-green, canary releases

Popular CI/CD Tools

Jenkins

  • Open-source, self-hosted
  • Highly customizable with plugins
  • Large community and ecosystem
  • Pipeline as code (Jenkinsfile)

GitLab CI

  • Integrated with GitLab repository
  • YAML-based configuration
  • Built-in container registry
  • Auto DevOps features

GitHub Actions

  • Native GitHub integration
  • Marketplace with reusable actions
  • Matrix builds
  • Easy secrets management

Others

  • CircleCI, Travis CI
  • Azure DevOps
  • AWS CodePipeline
  • Tekton, ArgoCD

Jenkins Pipeline Example

// Jenkinsfile
pipeline {
    agent any

    stages {
        stage('Build') {
            steps {
                sh 'npm install'
                sh 'npm run build'
            }
        }

        stage('Test') {
            steps {
                sh 'npm test'
            }
        }

        stage('Deploy') {
            when { branch 'main' }
            steps {
                sh './deploy.sh'
            }
        }
    }
}

GitLab CI Example

# .gitlab-ci.yml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/

test:
  stage: test
  script:
    - npm test

deploy:
  stage: deploy
  script:
    - ./deploy.sh
  only:
    - main

Automated Testing Strategies

Testing Pyramid

  • Unit Tests (Base): Fast, isolated, test individual functions
  • Integration Tests (Middle): Test component interactions
  • E2E Tests (Top): Full user workflows, slower but comprehensive

Additional Testing Types

  • Static Analysis: Linting, code quality checks
  • Security Scanning: SAST, dependency vulnerability checks
  • Performance Testing: Load testing, stress testing
  • Smoke Tests: Quick sanity checks after deployment

CI/CD Best Practices

Do

  • Commit code frequently (at least daily)
  • Keep builds fast (< 10 minutes ideal)
  • Fix broken builds immediately
  • Use feature branches and pull requests
  • Automate everything possible
  • Version your pipeline configuration

Avoid

  • Long-lived feature branches
  • Manual steps in the pipeline
  • Skipping tests to deploy faster
  • Deploying on Fridays (when starting out)
  • Ignoring flaky tests
  • Storing secrets in code

Remember: The goal is to make deployments boring, routine events - not stressful, risky operations.

Day 2

Infrastructure as Code

What is Infrastructure as Code?

Managing and provisioning infrastructure through machine-readable definition files rather than manual processes.

Traditional (Manual)

  • Click through web consoles
  • SSH into servers to configure
  • Document steps in wikis
  • Configuration drift over time
  • Hard to replicate environments

Infrastructure as Code

  • Define infrastructure in code files
  • Version control all changes
  • Automated provisioning
  • Consistent, repeatable deployments
  • Self-documenting infrastructure

Key Principle: Treat infrastructure the same way you treat application code - version it, review it, test it.

Benefits of Infrastructure as Code

Consistency

  • Same config = same result
  • Eliminate configuration drift
  • Reproducible environments

Speed

  • Provision in minutes
  • Clone environments easily
  • Rapid disaster recovery

Safety

  • Code review changes
  • Rollback capability
  • Audit trail via Git

Additional Benefits

  • Cost Savings: Automate repetitive tasks, reduce human error
  • Documentation: Code IS the documentation
  • Testing: Validate infrastructure before deployment
  • Collaboration: Teams can contribute and review changes

IaC Approaches

Declarative

  • Define desired end state
  • Tool figures out how to achieve it
  • Idempotent by design
  • Examples: Terraform, CloudFormation
# Declare what you want
resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t2.micro"
}

Imperative

  • Define step-by-step commands
  • Explicit control over process
  • Must handle idempotency
  • Examples: Ansible, Scripts
# Define how to do it
- name: Install nginx
  apt:
    name: nginx
    state: present
- name: Start nginx
  service:
    name: nginx
    state: started

Introduction to Terraform

Terraform is an open-source IaC tool by HashiCorp that enables you to define and provision infrastructure using a declarative configuration language.

Key Concepts

  • Providers: Plugins that interact with APIs (AWS, Azure, GCP, etc.)
  • Resources: Infrastructure components (instances, networks, databases)
  • State: Terraform's record of managed infrastructure
  • Plan: Preview changes before applying
  • Apply: Execute the planned changes

Terraform Example

# main.tf - Define an EC2 instance

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
}

resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name        = "WebServer"
    Environment = "Development"
  }
}

Essential Terraform Commands

# Initialize working directory (download providers)
terraform init

# Format configuration files
terraform fmt

# Validate configuration syntax
terraform validate

# Preview changes (dry run)
terraform plan

# Apply changes to infrastructure
terraform apply

# Show current state
terraform show

# Destroy infrastructure
terraform destroy

Tip: Always run terraform plan before apply to review changes!

Version Control for Infrastructure

Why Version Control Infrastructure Code?

  • History: Track all changes over time
  • Collaboration: Multiple team members can contribute
  • Review: Pull requests for infrastructure changes
  • Rollback: Revert to previous working state
  • Audit: Who changed what and when

Best Practices

  • Use meaningful commit messages
  • Branch protection for main/production
  • Require code reviews for infrastructure changes
  • Never commit secrets or credentials
  • Use .gitignore for state files and sensitive data

Credential Management

Securing Secrets in CI/CD Workflows

Credential Management Challenges

Common Problems

  • Secrets hardcoded in source code
  • Credentials stored in plain text config files
  • Shared credentials across environments
  • No rotation policy for secrets
  • Secrets visible in CI/CD logs

Real-World Impact: Exposed credentials are one of the most common causes of security breaches. One leaked API key can compromise entire systems.

Secrets Management Solutions

CI/CD Platform Secrets

  • GitHub Secrets
  • GitLab CI Variables
  • Jenkins Credentials
  • Environment variables (masked)

Dedicated Tools

  • HashiCorp Vault
  • AWS Secrets Manager
  • Azure Key Vault
  • CyberArk, 1Password

Best Practices

  • Never commit secrets to version control
  • Use environment-specific credentials
  • Implement least privilege access
  • Rotate secrets regularly
  • Audit secret access
  • Use short-lived credentials when possible

Using Secrets in Pipelines

GitHub Actions Example

name: Deploy
on: push

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Deploy to Server
        env:
          # Reference secrets from repository settings
          API_KEY: ${{ secrets.API_KEY }}
          DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
        run: |
          ./deploy.sh

Note: Secrets are automatically masked in logs. Never echo or print secret values!

Containerization & Kubernetes

What are Containers?

Containers are lightweight, standalone packages that include everything needed to run an application: code, runtime, libraries, and system tools.

Virtual Machines

  • Full OS per instance
  • Gigabytes in size
  • Minutes to start
  • Hardware-level isolation
  • More resource intensive

Containers

  • Shared OS kernel
  • Megabytes in size
  • Seconds to start
  • Process-level isolation
  • Lightweight and efficient

Key Benefit: "It works on my machine" â†’ "It works in this container" - Consistent behavior across all environments.

Introduction to Docker

Docker is the most popular containerization platform, making it easy to build, ship, and run containers.

Key Concepts

  • Image: Read-only template with instructions for creating a container
  • Container: Running instance of an image
  • Dockerfile: Script that defines how to build an image
  • Registry: Storage for Docker images (Docker Hub, ECR, etc.)
  • Volume: Persistent storage for container data

Writing a Dockerfile

# Use an official base image
FROM node:18-alpine

# Set working directory
WORKDIR /app

# Copy package files first (for caching)
COPY package*.json ./

# Install dependencies
RUN npm install --production

# Copy application code
COPY . .

# Expose port
EXPOSE 3000

# Define startup command
CMD ["node", "server.js"]

Essential Docker Commands

# Build an image from Dockerfile
docker build -t myapp:v1 .

# List images
docker images

# Run a container
docker run -d -p 3000:3000 --name myapp myapp:v1

# List running containers
docker ps

# View container logs
docker logs myapp

# Execute command in container
docker exec -it myapp /bin/sh

# Stop and remove container
docker stop myapp && docker rm myapp

# Push image to registry
docker push myregistry/myapp:v1

Docker Best Practices

Image Building

  • Use official base images
  • Use specific version tags, not :latest
  • Minimize layers, combine RUN commands
  • Order instructions for caching
  • Use multi-stage builds
  • Don't run as root

Security

  • Scan images for vulnerabilities
  • Don't store secrets in images
  • Use .dockerignore file
  • Keep images small (Alpine base)
  • Update base images regularly
  • Use read-only file systems when possible

Introduction to Kubernetes

Kubernetes (K8s) is an open-source container orchestration platform that automates deployment, scaling, and management of containerized applications.

Why Kubernetes?

  • Orchestration: Manage thousands of containers across multiple hosts
  • Scaling: Automatically scale applications based on demand
  • Self-Healing: Restart failed containers, replace unhealthy nodes
  • Service Discovery: Built-in DNS and load balancing
  • Rollouts: Zero-downtime deployments and rollbacks

Note: Kubernetes manages containers at scale. Docker runs containers; Kubernetes orchestrates them.

Kubernetes Architecture

Control Plane

  • API Server: Frontend for K8s
  • etcd: Key-value store for cluster data
  • Scheduler: Assigns pods to nodes
  • Controller Manager: Manages cluster state

Worker Nodes

  • kubelet: Ensures containers are running
  • kube-proxy: Network proxy for services
  • Container Runtime: Docker, containerd, etc.

Key Objects

  • Pod: Smallest deployable unit (one or more containers)
  • Deployment: Manages pod replicas and updates
  • Service: Network endpoint to access pods
  • ConfigMap/Secret: Configuration and sensitive data

Pods

A Pod is the smallest deployable unit in Kubernetes - a group of one or more containers that share storage and network.

# pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    app: web
spec:
  containers:
  - name: web
    image: nginx:1.24
    ports:
    - containerPort: 80
# Create pod from YAML
kubectl apply -f pod.yaml

# View pods
kubectl get pods

Deployments

Deployments manage ReplicaSets and provide declarative updates, scaling, and rollback capabilities.

# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: nginx:1.24
        ports:
        - containerPort: 80

Services

Services provide stable network endpoints to access pods, with built-in load balancing.

# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 80
  type: LoadBalancer

Service Types

  • ClusterIP: Internal cluster access only (default)
  • NodePort: Expose on each node's IP at a static port
  • LoadBalancer: Expose via cloud load balancer

Essential kubectl Commands

# Apply configuration
kubectl apply -f deployment.yaml

# Get resources
kubectl get pods
kubectl get deployments
kubectl get services

# Describe resource details
kubectl describe pod my-app

# View logs
kubectl logs my-app

# Execute command in pod
kubectl exec -it my-app -- /bin/sh

# Scale deployment
kubectl scale deployment my-app --replicas=5

# Delete resources
kubectl delete -f deployment.yaml
